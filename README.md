# File-transfer

شبكة نقل الملفات المحلية — تطبيق خفيف الوزن قائم على Node.js يسمح بنقل ملفات مباشرة بين الأجهزة المتصلة بنفس الشبكة عبر واجهة ويب (P2P)، مع ميزات مساعدة مثل توليد رموز QR، كشف captive-portal، وتجميع أجزاء الملفات على الخادم.

## المزايا الرئيسية
- نقل مباشر بين الأجهزة على الشبكة المحلية (لا يتطلب إنترنت لنقل الملفات).
- صفحة عميل ويب جاهزة: `client-p2p.html` لتجربة المستخدم وإرسال/استقبال الملفات.
- خادم Node.js مركزي (`server.js`) لخدمات الإشارة، توليد QR، وتخزين الملفات الواردة في مجلد `downloads/`.
- نقطة نهاية لتوليد وعرض رمز QR (`/generate-qrcode`, `/qrcode-page`).
- إدارة تجزئة الملفات (chunks) وتجميعها تلقائياً على الخادم.
- أدوات اختبار وتحسين السرعة: `speed-test.js` و `speed-optimizations.js`.

## المكدس (Stack)
- لغة: JavaScript (Node.js)
- إطار العمل: Express + Socket.IO
- توليد QR: qrcode

## متطلبات التشغيل
- Node.js v14+ (موصى به)
- npm

## التهيئة السريعة
1. انسخ المستودع:

```bash
git clone https://github.com/sdevp-tech/File-transfer.git
cd File-transfer
```

2. ثبت الاعتمادات:

```bash
npm install
```

3. شغّل الخادم:

```bash
node server.js
```

- افتح المتصفح وانتقل إلى `http://localhost:3000` أو إلى العنوان الذي يطبع في سجل الطرفية (مثل `http://<IP>:3000/client-p2p.html`).
- لتغيير المنفذ استخدم متغير البيئة `PORT`:

```bash
PORT=8080 node server.js
```

ملاحظة: الخادم يحاول تشغيل بوابة Captive Portal على المنفذ 80. إن لم تكن تملك صلاحيات root سيطبع إرشادات بديلة (استخدام sudo أو إعادة توجيه iptables).

## نقاط النهاية المفيدة
- /           → صفحة بوابة بدأت التصميم تلقائياً وتعيد التوجيه إلى `client-p2p.html` مع عرض QR.
- /client-p2p.html → واجهة العميل (static file).
- /generate-qrcode → توليد رمز QR (يمكن طلب PNG أو صفحة HTML). مثال: `/generate-qrcode?format=png&size=300&url=<URL>`
- /qrcode-page → صفحة كاملة تعرض QR ورابط التطبيق.
- /downloads   → مجلد الاستضافة لملفات التحميل (static).

## التفاصيل الفنية (ملخص سريع من server.js)
- يستخدم Socket.IO لتسجيل المستخدمين وتبادل إشعارات بدء/قبول/إرسال القطع (chunks).
- الملفات تُستقبل كقطع Base64 وتُكتب مؤقتاً إلى `temp_chunks/` ثم تُجمع في `downloads/`.
- توجد آليات لتنظيف القطع والـ transfers المنتهية زمنياً.
- توجد قائمة `portalEndpoints` لدعم اكتشاف شبكات Captive Portal وإعادة التوجيه إلى صفحة التطبيق.

## نصائح أمان
- تأكد من ضبط قيود على حجم الملف المسموح به (حماية من استنزاف القرص).
- تحقق من أنواع الملفات إذا أردت منع ملفات خطرة.
- لتشغيل في بيئة إنتاج استخدم HTTPS (HTTPS مطلوب في أغلب المتصفحات لميزات WebRTC المتقدمة).
- حدّث التبعيات بانتظام (npm audit).

## إعدادات مقترحة للإنتاج
- فصل خدمة الإشارة (Signaling) إلى خدمة WebSocket/Redis منفصلة إذا زاد عدد المستخدمين.
- تشغيل خلف Nginx مع TLS وتهيئة البروكسي العكسي.
- مراقبة السجلات ومسح دوري للمجلد `temp_chunks/` و`downloads/` وفق سياسات الاحتفاظ.

## كيفية المساهمة
- استخدم الـ branch workflow: fork → feature branch → pull request مع وصف واضح.
- لا تدفع `node_modules/` داخل PRs ولا تُعدّل `package-lock.json` يدوياً إلا عند تحديث تبعيات.

## استكشاف الأخطاء الشائع
- التطبيق لا يستقبل اتصالات: تأكد من أن المنفذ مفتوح والفِيروال يسمح بالاتصال داخل الشبكة.
- فشل توليد QR: تأكد من أن الخدمة تستطيع الوصول إلى عنوان IP الصحيح على الشبكة المحلية.
- ملفات ناقصة في الواجهة: افتح `client-p2p.html` عبر الخادم وليس مباشرة من نظام الملفات لتتجنب مشاكل CORS وSignaling.
